# 语音交互设计蓝图

Date: 2026-06-19

## 核心原则

- **Voice-first but not voice-only**：语音是主要输入通道，但所有操作都有视觉反馈（屏幕指示器、状态栏、浮动面板）。用户应能全程通过视觉确认当前状态。
- **语音指令映射到 MCP 工具**：自然语言指令经过意图识别后，最终调用 `MCPToolRegistry` 中已注册的 94+ 工具执行。不存在独立的"语音指令解析器"——指令分类由上层意图识别模块处理，执行层复用现有的工具链和安全层。
- **分层确认策略**：根据操作风险等级（inherit from `ActionEngine.riskLevel`）决定是否需要语音确认或二次确认，高风险操作自动激活 `highRisk` 模式拦截。

## 语音指令分类

### 导航类（无风险，直接执行）
用户说出导航或观察指令，系统直接执行并返回视觉结果。

| 语音指令 | 映射工具 | 说明 |
|---|---|---|
| "看屏幕" | `screen_context` | 触发截图 + OCR，返回文字描述 |
| "点保存按钮" | `click` | 无确认，视觉反馈高亮点击位置 |
| "打开 Safari" | `keyboard` + `type` | 模拟 Cmd+Space，输入应用名 |
| "滚动到下面" | `scroll` | 模拟滚轮事件 |
| "现在开了什么窗口" | `get_app_state` | 返回活跃窗口信息 |

### 查询类（低风险，返回后语音朗读）
查询系统状态或读取信息，工具执行后通过 TTS 朗读结果，同时显示在浮动面板。

| 语音指令 | 映射工具 | 说明 |
|---|---|---|
| "当前是什么模式" | `get_app_state` / modes | TTS 告知模式状态 |
| "看看审计日志" | 读取审计文件 | 朗读摘要，面板显示全文 |
| "这个窗口是什么" | `get_app_state` | 返回窗口标题和 AX 树摘要 |

### 操作类（高风险，需要确认）
涉及对 UI 控件或文件系统的修改操作。执行前必须语音确认，确认后仍走 `ActionEngine` 安全评估。

- **单步操作**（如"点发送"、"关闭标签页"）：系统语音反问"确认点击发送按钮？"，用户说"确认"或"是"后执行。提供 5 秒超时，超时自动取消。
- **多步操作**（如"删掉 Downloads 里的 .tmp 文件"）：进入 `ExecutionPlan` 流程，生成 1-3 步计划，用户逐条确认或一键批准全部。

### 安全类（受 ModeManager 控制）
安全指令直接调用 `ModeManager` API，受 `policyLock` 保护。被锁定的模式无法通过语音解除。

| 语音指令 | 模式切换 | 受保护 |
|---|---|---|
| "切换到只读模式" | `modeManager.activate(.readOnly)` | policyLock 时可拦截 |
| "开启审计导出" | `auditExport` mode | policyLock 时可拦截 |
| "解锁只读模式" | `modeManager.deactivate(.readOnly)` | policyLock 时拒绝 |
| "锁定策略" | `policyLock` mode | 仅管理员操作 |

## 语音反馈设计

对话式三段反馈，所有阶段都同步更新视觉状态：

### 阶段 1：确认（高风险操作专属）
- 语音："确认点击发送按钮？"
- 视觉：浮动面板显示确认卡片，含操作描述和风险等级标签
- 状态指示器切换为黄色（处理中）
- 用户可语音确认或说"取消"；5 秒超时自动取消

### 阶段 2：执行中
- 语音：简短状态提示（"正在操作..."）
- 视觉：浮动面板显示进度条或旋转指示器；当前操作名称显示在状态栏
- 状态指示器保持黄色

### 阶段 3：结果
- **成功**：语音播报结果摘要（"已完成：已点击发送按钮"）+ 截图证据（如适用）+ 状态指示器回归绿色
- **失败**：语音播报错误（"操作失败：窗口未找到"）+ 建议替代方案（"是否要尝试搜索发送按钮？"）+ 视觉显示错误详情

### 取消路径
用户随时可说"停止"、"取消"、"算了"——`cancelVoiceInput()` 中断当前语音捕获和 TTS 播放，回到监听状态。

## 状态指示器

浮动面板左上角始终显示圆形状态灯，颜色语义如下：

| 状态 | 含义 |
|---|---|
| 绿色 | 正常监听，等待语音输入 |
| 黄色 | 处理中（正在识别/执行操作/朗读响应） |
| 红色 | 高风险操作待确认（闪烁） |
| 灰色 | 空闲/语音输入已暂停 |

## 与现有架构的关系

本设计不引入新的独立语音引擎。语音输入复用 `RenJistrolySystemBridge.MacOSSpeechRecognizer`（Round 04），语音输出复用 `MacOSTextToSpeech`（Round 05），指令执行复用 `MCPToolRegistry` + `ActionEngine` + `ModeManager`（Round 07 Enterprise）。新增的仅是**意图到工具的映射层**和**语音确认对话流**——两者均可由一个轻量 `VoiceInteractionCoordinator` 实现，挂载到 `ConversationEngine` 中作为语音输入流的后处理管道。
